# DataVision Pro — Marketing Site

A static, fully crawlable marketing site for DataVision Pro, built with [Astro](https://astro.build/). It lives **alongside** the Streamlit app (which stays as the product) and is what search engines and AI engines (ChatGPT, Perplexity, Google AI Overviews) actually see.

## Why a separate site

Streamlit renders content via WebSocket / JS after page load, so search engines see an empty shell. This site is server-rendered HTML with full content, JSON-LD, and Open Graph tags in the response — `curl` can see every word.

## Stack

- **Astro 4** — zero JS by default, ideal for SEO/GEO content.
- **Custom `/sitemap.xml`** endpoint at `src/pages/sitemap.xml.ts` — auto-includes all glossary, compare, and guide entries.
- Plain CSS, no Tailwind, no UI framework — keeps Lighthouse fast.

## Run locally

```bash
cd marketing-site
npm install   # only the first time
npm run dev   # http://0.0.0.0:8000
```

Or via Replit workflow: the **Marketing site** workflow runs `astro dev` on port 8000.

## Build for production

```bash
npm run build       # outputs ./dist
npm run preview     # serve dist on :8000
```

## Deploy

The build output (`marketing-site/dist`) is fully static and can be hosted on:

- Replit Static Deployment (recommended — separate from the Streamlit autoscale deployment)
- Cloudflare Pages, Netlify, Vercel, GitHub Pages — any static host

Set the env var `APP_URL` at build time to point the "Launch App" / "Sign In" buttons at the live Streamlit URL (defaults to `https://app.datavisionpro.app`):

```bash
APP_URL=https://your-streamlit-domain.replit.app npm run build
```

### Publishing on Replit (step-by-step)

The project's `.replit` `[deployment]` block is configured for a **Static** deployment of this site:

```toml
[deployment]
deploymentTarget = "static"
build = ["bash", "-c", "cd marketing-site && npm install && npm run build"]
publicDir = "marketing-site/dist"
```

> ⚠️ The Streamlit product app (port 5000) lives in a **separate** Replit deployment (autoscale, `streamlit run app.py --server.port 5000`) at `https://app.datavisionpro.app`. Do not collapse the two — the marketing site must stay statically hosted so Google, Bing, and AI crawlers can read the server-rendered HTML.

**Steps:**

1. Open the **Publishing** tool in Replit on the *main* version of the project (after this task is merged).
2. Confirm the Static deployment settings shown above appear pre-filled.
3. Click **Publish**. Replit runs the `build` command and serves `marketing-site/dist` over HTTPS at a `*.replit.app` URL.
4. In the deployment's **Settings → Domains**, add the custom domain `datavisionpro.app` (and `www.datavisionpro.app` if desired). Replit will show DNS records (`A` / `AAAA` / `TXT`) — add them at your domain registrar. HTTPS certs are issued automatically once DNS resolves.
5. Once the domain is live, verify these URLs return 200 over HTTPS:
   - `https://datavisionpro.app/` (home)
   - `https://datavisionpro.app/robots.txt` (must list `Sitemap: https://datavisionpro.app/sitemap.xml`)
   - `https://datavisionpro.app/sitemap.xml` (XML urlset listing all 22 pages, generated by `src/pages/sitemap.xml.ts`)
   - A sample of glossary / compare / guide slugs from that sitemap.

### Submit to search engines

Once `https://datavisionpro.app` resolves with HTTPS:

1. **Google Search Console** → <https://search.google.com/search-console> → *Add property* → URL prefix `https://datavisionpro.app/` → verify (DNS TXT record is easiest because the domain is already on Replit) → *Sitemaps* → submit `sitemap.xml`.
2. **Bing Webmaster Tools** → <https://www.bing.com/webmasters> → *Add a site* → `https://datavisionpro.app/` → verify (you can import directly from Google Search Console) → *Sitemaps* → submit `https://datavisionpro.app/sitemap.xml`.
3. (Optional) **IndexNow**: Bing/Yandex auto-discover via the submitted sitemap, but you can speed up the first crawl with an IndexNow ping.

After submission, expect first indexing within a few days; full crawl of all 22 pages typically completes within ~2 weeks.

### Auto-pinging on every deploy (IndexNow + Google Indexing API)

The build is wired to a `postbuild` hook that runs `scripts/ping-search-engines.mjs`. After every successful `npm run build` (which is what the Replit Static deployment runs), the script:

1. Reads the freshly generated `marketing-site/dist/sitemap.xml` (or `sitemap-index.xml` if present).
2. Diffs the URLs and `lastmod` values against `marketing-site/.indexnow-state.json` (committed, so first-run vs. ongoing-run is deterministic across deploys).
3. POSTs only the **new or changed** URLs to **IndexNow** (Bing, Yandex, Seznam, Naver) in a single batch request.
4. Optionally POSTs each URL to the **Google Indexing API** if a service-account credential is configured.
5. Writes the new state and logs every URL it pinged.

Failures never break a deploy — the script always exits `0` and logs warnings.

#### One-time setup — IndexNow

1. Generate a key (32+ hex chars). Easiest:
   ```bash
   node -e "console.log(require('crypto').randomBytes(16).toString('hex'))"
   ```
2. Create a key file in `marketing-site/public/` named `<your-key>.txt` whose contents are exactly the key string. Astro copies `public/` verbatim into `dist/`, so it will be served at `https://datavisionpro.app/<your-key>.txt` after deploy. Verify with `curl`.
3. Set the env var on the **Static deployment** in Replit (Deployments → Settings → Secrets):
   - `INDEXNOW_KEY` = the same key string
   - (Optional) `INDEXNOW_KEY_LOCATION` = override the default `https://<host>/<key>.txt` location.

If `INDEXNOW_KEY` is not set the script logs a warning and skips IndexNow — no failure.

#### One-time setup — Google Indexing API (optional)

> Heads-up: Google officially supports this API only for `JobPosting` and `BroadcastEvent` pages. Submitting other URLs is allowed but Google may ignore them. We still expose the hook for completeness and for the day a job-board / event page ships.

1. In Google Cloud Console, create a project, enable the **Indexing API**, create a **service account**, and download its JSON key.
2. In Google Search Console, open the property for `https://datavisionpro.app/`, **Settings → Users and permissions → Add user**, paste the service account's `client_email`, give it **Owner**.
3. Set the env var on the Static deployment:
   - `GOOGLE_INDEXING_SERVICE_ACCOUNT_JSON` = the **entire JSON file contents** as a single string.

If unset, the script logs a line and skips Google — no failure.

#### Manual run / debugging

```bash
# After a build, from the project root:
DRY_RUN=1 node scripts/ping-search-engines.mjs   # logs what it would ping, sends nothing
node scripts/ping-search-engines.mjs             # actually pings
```

To force a full re-ping of every URL, delete `marketing-site/.indexnow-state.json` and run the script again.

## Content rules

These are non-negotiable and apply to every new page:

1. **Information gain.** Each page must contain at least one piece of insight or data not present in the current top-3 Google results for the target query.
2. **No fabrication.** No invented testimonials, ROI percentages, customer names, or competitor metrics. Where a number is needed and unavailable, leave a clearly-marked `[INSERT: …]` placeholder.
3. **Source every stat.** All cited statistics link to a primary source (research paper, government / .edu publication, or the vendor's own published materials).
4. **GEO structure.** First 40–60 words = direct answer. Then stats with citations, then how-it-works, then FAQ, then related links.
5. **No thin pages.** ~25 high-quality pages beats 200 thin ones.

## Editing content (no code required)

All marketing content lives as plain Markdown files with YAML frontmatter under `src/content/`. Non-developers can add or edit pages without touching any TypeScript or Astro code — just drop a new `.md` file in the right folder, fill in the frontmatter, and the site picks it up on the next build.

### Folder layout

```
src/content/
├── config.ts                # Schema (validates frontmatter on build — do not edit unless adding a field)
├── glossary/<slug>.md       # One file per glossary term  → /glossary/<slug>
├── compare/<slug>.md        # One file per competitor     → /compare/<slug>
└── guides/<slug>.md         # One file per how-to guide   → /guides/<slug>
```

The filename (minus `.md`) becomes the URL slug. The build will fail loudly if any required field is missing or has the wrong type, so typos can't ship silently.

### Adding a new glossary term

Create `src/content/glossary/my-new-term.md`:

```markdown
---
term: "My New Term"
question: "What is my new term?"
shortDef: "One-sentence definition that appears on cards."
description: "Meta description for search engines (~155 chars)."
answer: "40–60 word direct answer that opens the page."
stats:
  - value: "42%"
    label: "Why this number matters."
    source:
      label: "Source name"
      url: "https://example.com/source"
faq:
  - q: "First question?"
    a: "Direct answer."
related:
  - "data-cleaning"
  - "outlier-detection"
updated: "2026-04-21"
---

## How it works

Write the body in **Markdown**. Inline HTML (`<ol>`, `<table>`, etc.) also works.

## Why it matters

Each `## Heading` becomes a section on the page.
```

### Adding a new compare page

Create `src/content/compare/datavision-pro-vs-<competitor>.md`. All content for compare pages lives in the frontmatter (no markdown body needed):

```markdown
---
competitor: "Looker"
title: "DataVision Pro vs Looker"
description: "Honest, side-by-side comparison."
intro: "40–60 word direct answer."
bestFor:
  us: "Who DataVision Pro is best for."
  them: "Who Looker is best for."
rows:
  - feature: "Setup time"
    us: "Minutes"
    them: "Days"
whenToChoose:
  us:
    - "Bullet list of when DataVision Pro wins."
  them:
    - "Bullet list of when Looker wins."
faq:
  - q: "Question?"
    a: "Answer."
updated: "2026-04-21"
---
```

### Adding a new guide

Create `src/content/guides/how-to-do-something.md`. Each `## Heading` in the body becomes one ordered step in the page **and** in the `HowTo` JSON-LD schema for search engines, so the heading order is the step order:

```markdown
---
title: "How to do something useful"
description: "One-line meta description."
intro: "40–60 word direct answer."
estTime: "5 minutes"
difficulty: "Beginner"           # Beginner | Intermediate | Advanced
prerequisites:
  - "What you need before starting."
pitfalls:
  - "Common mistakes to avoid."
faq:
  - q: "Question?"
    a: "Answer."
updated: "2026-04-21"
---

## Step name one

What to do. Markdown plus inline HTML both work.

## Step name two

Next step.
```

### Editing an existing page

Open the `.md` file under `src/content/<collection>/<slug>.md`, edit the frontmatter or the body, and save. The dev server hot-reloads; production picks up the change on the next build.

### Validation

Frontmatter is validated against the Zod schema in `src/content/config.ts` at build time. If you forget a required field or use the wrong type, the build fails with a clear error pointing at the file and field — nothing broken can ship.

## Crawlability check

```bash
# After running `npm run preview` (or against the live site)
curl -s http://localhost:8000/glossary/data-cleaning | grep "What is data cleaning?"
curl -s http://localhost:8000/ | grep "DataVision Pro"
```

Both should print matching lines. If they don't, the page is not server-rendered and you've broken the contract.
